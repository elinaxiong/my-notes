  1. 上下文是新的瓶颈，不是模型能力。千万行代码、上亿 Token 的仓库不能"一次喂完"，必须按目录、领域、章节做正交拆解，再用代理按章生成。
  2. 大模型长上下文 ≠ 全精度上下文。200K~1M 的窗口在 500K 之后准确率显著下滑，DeepSeek V4 / 4.7 OF 等评测显示长上下文仍以"搜索任务"为训练范式，做生成时需谨慎使用。
  3. Agent 工程的主战场，是给模型"更准上下文 + 更稳执行环境"。Loop 代码已从万行回落到千行级，焦点转向工具、记忆、回放与回滚。
  4. "自然语言驱动 + Multi-Agent 编排"是中后台流程改造的可行路径。李伟把 3~5 天、跨 3 个团队的字段变更，压到 2~3 小时一句话完成，每天 token 成本仅 10 余元。
  5. 压测是负载建模，不是请求构造。真实性、覆盖度、复杂度、有效性四难，决定了压测结果是否可被信任；过度拟合线上 = 低估未来风险。
  6. 测试 Agent 的关键，不在 prompt 技巧，而在业务知识库结构 + 调试经验沉淀。RAG 不适合测试用例规划——召回不全就意味着漏测。
  7. 记忆让 Agent 越用越聪明。知识库（领域、接口、知识）+ 脚本经验（成功路径、踩坑沉淀）双库联用，10~20 个用例后新用例的一次成功率显著抬升。

一、AI Coding 两年范式迁移：从提示词到 Agentic

  • 24 年 · 提示词工程。圈选 20 行代码 → 让模型解释 → 单次拼接、强幻觉，很快触顶。
  • 25 年初 · 上下文工程 + Code Review。用 AST / Transformer 拼接函数上下游，再交给模型；同期代码补全、代码审查同时火热。
  • 25 年中 · Coding Agent 成为主力。Claude / Local Code 类 Agent 进入视野，Loop 代码从上万行降到千行级；工作重心彻底转向"给模型更准上下文 + 更稳执行环境"。
  • 26 年 · 通用 Coding Agent 趋近 SWE-Bench 饱和。Anthropic Lighthouse 等模型已"摸到 API 边界"；研发需求从"补全/审查"升级为"端到端交付的数字员工"。

二、为什么"上下文是被吃掉的"——一笔账

讲者把"用 200K 上下文模型理解一个 60 万行 Android 仓库"作为压力测试，逐项拆解上下文消耗结构：

  • 代码本身。业务代码 ≈ 15 token / 行，10 行代码就吃掉 200K 上下文的可见量级；100 章节 × 50~100K → 不做优化、纯靠模型完全跑不动。
  • 文件路径——隐形消耗大户。一行 Java/Python 路径就要 30~50 token。早期 Local Code / Code Plugin 直接展开整棵目录树，"2/3 的上下文被路径吃掉"。
  • 真实数字。小红书 Android 仓 60 万文件，光文件名 ≈ 180 万 token，远超任何模型上下文。

关于"长上下文 ≠ 全精度上下文"

讲者当周与 Gemini 长上下文核心开发者 Andrew Guy 在中国大陆交流。结论是：当前所有 1M 级长上下文模型，前段精度高、500K~600K 中段精度显著下滑。原因有二：MRCR / 大海捞针评测只能验证"末尾召回"，且训练任务以搜索为主，奖励反馈来自搜索准确性。

典型案例：DeepSeek V3.2 凭 NSA + MHA KV-Cache 压缩把长上下文成本打到极低，团队第一时间尝试"把所有候选直接塞 1M 上下文每次追问"，实际不可行；DeepSeek V4 长上下文精度也下降了，原因之一是为压成本减少了 MoE 激活专家。

三、CodeWiki 核心策略：从"读完仓库"到"正交拆解"

面对 60 万行、200K 上下文的不匹配，方案是不做一次性整仓阅读，而是拆成足够正交、足够完整的小任务，逐个击破。整体落到两阶段。

阶段 1 · 大纲生成（战术师视角）

  • 用粗粒度阅读 + 不断分片，把仓库拆成"50 片左右、每片 ≈ 5000 文件"。
  • 大纲 Agent 看更多、看更远，不下钻细节，逐层展开形成全局结构与章节列表。
  • 大纲生成阶段成本被压到全流程的 ~10%，"节省 90% 成本"主要靠这里。

阶段 2 · 内容生成（按章节启子 Agent）

  • 每个章节 = 一个独立子智能体启动指令；子 Agent 内部递归拆图、生成、合并。
  • 关键技巧：压缩用于探索期，因果递归用于真正写内容时——让模型在写作阶段保持在同一表达空间。
  • 抽取与映射靠"模板 + 实体"，企业级仓库本身目录划分相对规整，可大幅减少模型的盲猜。

工程化细节

  • 使用 rank + batch 两类工具——对应 DeepSeek 在 SWE-Bench 技术报告里"接近 80 分"的实测组合。
  • 判断项 / 辩论项全部以代词形式注入，避免 "Home dash" 类词被模型直接捕获造成方向偏差。
  • 地理 / 业务向量等"非代码信息"也作为子节点辅助上下文（业内有公开案例：相关请求的相关门店大概率落在同一地理区域）。

四、产品成果 · 周期与成本

  • 覆盖小红书全部"热门商户"（仓）级代码库，周期性全量更新一次。
  • 整月成本约 ¥300。给所有热门仓生成结构化知识文档，"无论从覆盖还是成本看都很划算"。
  • 可同时服务"人快速上手陌生仓"和"AI 拿来作为代码理解输入"两类消费者。

五、个人观点 · 长程任务作为团队准入门槛

讲者认为"代码仓库 → Wiki" 适合作为团队 / 阶段性能力的考题：评估生成对不对、接口文档准不准，一眼可看；任务要素丰富——多智能体、长时序、稳定运行、上下文压缩与裁剪缺一不可。"如果你是一个团队，这东西非常适合作为准入门槛。"

一、痛点 · 一次"加字段"的真实代价

广告链路三层架构：B 端投放端 → 数据管道 → C 端在线层。每一次字段变更都涉及：

  • 跨团队人传人。口头约定、群里沟通、命名 / 大小写不一致 → 端到端"失真"。
  • 影响面靠经验判断。把一个 int8 扩到 int64，下游所有服务是否同步扩容，无人能给出确定答案。
  • 追溯断裂。字段含义、当初为什么改、谁改的，半年后已无人可问。
  • 验证零散。每端各自验，端到端在群里"靠相信"，无统一验收标准。

二、目标态：一句话驱动的全链路自动化

研发在企业 IM 群里和 Agent 对话——表达一句"我要在 X 字段上加 Y 含义"——Agent 自动理解意图、识别影响面、串联各业务域子 Agent 完成执行、按节点收集人工审核、最终产出含背景 / 对话 / 验证结果的变更知识链报告。

三、产品架构 · 主从双层 Agent

层角色关键能力
主引擎 Actor Engine监督 + 指挥官意图理解 / 编排调度 / 影响面升级 / 人工卡控
业务域子 Agent实际执行者投放端 / 数据管道 / 在线索引各一个 SCP 子引擎，按 Skill 注册到主引擎
能力底座对接厂内系统变更平台、数据库、分布式配置、大数据任务、可观测、告警

选用厂内开源的 Java 框架（讲者称 aikm）作为基座——天然适配厂内服务，原生集成可观测、变更平台等基础设施。

四、四大关键挑战与解

挑战 1 · 字段血缘怎么自动维护？

  • 对中间数据管道做白盒分析：扫代码仓 + 阿尔法分布式配置 + 投放端库表结构 + 在线索引层使用方。
  • 每天凌晨全链路扫描一次校准，保证 B/C 端字段映射、上下游使用方在天级窗口内一致。
  • 有了血缘 → 自动产出"影响哪些表、哪些服务、各自 Owner 是谁"的影响评估报告。

挑战 2 · 流程刚性 · 红线动作不能跳

  • 核心动作（如发布上库、回滚高危）强制人工审核，研发"觉得流程麻烦"也不允许跳过。
  • 每一步状态机管理 + 工单状态机叠加；超时、失败、A/B 验收不通过时支持节点回退与重试。
  • 异常需要人工介入时及时上报，不允许"为了把流程跑快而让模型自作主张"。

挑战 3 · 长期记忆与知识沉淀

  • 群聊原始对话、各 Action 执行详情、确认 / 澄清记录全量留痕，按需求工单组织。
  • 每完成一次变更，自动回写血缘表 + 沉淀 Action 经验 → "越用越聪明"。

挑战 4 · 协议与通信

  • 使用厂内自定义的 SCP 协议（HTTP / Socket 流式），主引擎与子 Agent 之间维护 Session。
  • 同步任务流式回传；像分级发布这种持久任务则发单后定时轮询工单状态。
  • Skill 只定义能力约束（"你能做什么"），不定义实现，主引擎按需调用。

五、推广策略 · 如何让研发愿意用

  • 第一步：辅助模式。系统先生成变更方案给研发参考，不夺权；研发观察方案与自身经验贴合度。
  • 第二步：习惯转移。方案逐渐贴现，研发开始按 Agent 的方案走。
  • 第三步（激进）：权限回收。磨合一段时间后回收变更平台直接操作权限，倒逼上 Agent。

六、收益数据

  • 变更周期：3~5 天 → 2~3 小时，最快 1 小时起。
  • 变更遗漏 / 顺序错乱：降为零。
  • 全链路 100% 可追溯。
  • 每天 token 成本 ≈ 10 余元。

七、现场问答精选

  • Q · 跨部门隐性知识，Agent 怎么吸收？
    聚焦"数据链路"这个流程相对固定的子领域；隐性知识第一步靠字段血缘还原，第二步沉淀为 Action 知识，从而摆脱对个人经验的依赖。
  • Q · 字段一直在变，怎么保证 Agent 拿到的不是过期版本？
    变更本身走 Agent → 完成后自动回写血缘；同时凌晨全量扫一遍代码与配置做 T+1 校准。
  • Q · 整体 token 成本？
    每天 10 余元（千问类商用模型），相对人力跨团队协作的开销极低。
  • Q · 怎么评估推广效果？
    前期半智能辅助阶段建立信任，再回收变更平台权限做硬切换；本质是"流程改造 + AI 工具"组合，不是单纯加工具。
  • Q · SCP 协议在 Skill 与 Action 间怎么传 response？
    建会话拿 Session ID，再下发任务；同步走流式回传，异步走工单 + 轮询。
  • Q · 工作流任务超时 / 审批超时如何管理？
    底层是传统服务 + 状态机，主引擎负责跨节点状态流转，每一步可重试 / 回退到指定节点。

开场镜头 · 为什么压测开始失效

压测首轮上下文 20K，线上只有 12K；压测 thinking 占比 37%，线上只有 20%。数据错了，结论就没有意义。陈圣的演讲，就是讲怎么从"构造请求"走到"建模负载"。

一、传统压测 vs 模型压测

维度传统压测模型压测
输入结构固定（cursor / uid / note_id）不可枚举的自然语言
容量度量QPS / RT / 错误率TPS / 吞吐 / GPU 利用率
核心假设流量 ≈ 请求数流量 ≈ 内容分布与差分
风险面主链路饱和长尾、长上下文、多轮、工具调用

二、数据构造的"四难"

  1. 真实性难还原。线上输入有长尾、输出规模差异大、上下文层次不一；压测样本却整齐划一 → 高估存储、低估 TPS、长尾误判。
  2. 覆盖度难保障。新接业务往往只覆盖主链路、高频卡片；POI、外部依赖、复杂检索工具被漏掉，复杂度瓶颈不暴露。
  3. 复杂度难分层。不分层 → 压出问题不可解释（容量边界究竟来自哪种业务、哪类卡片？）。
  4. 有效性难判断。数据"可用 ≠ 结果可信"，需对发包率、异常样本、链路覆盖、回归一致性单独校验。

三、案例 · 点点 0.3.0 的两次迭代

第一版 · 传统流量录制

  • 从工程侧录线上流量 → 脱敏 UID / Session → 直接发压。
  • 结果：首轮上下文压出 20K（线上 12K），TTFB 压出 3s（线上 500ms），负载结构和线上完全错位。

第二版 · 引入"线上语料分布"模块

  • 显式提取线上语料分布、关键词，反推目标配比，让样本更贴线上。
  • 把笔记物料从"用户拥有的笔记"换成"用户真实读过的笔记"——通过视频 / 笔记底详接口反推 UID 拼接，避免"have 表式"虚假数据。
  • 分布对齐后，TTFB / 首发包出现"流量上涨反而下降"——分析后定位为新用户引导评论场景流量变多，上下文短、token 消耗低。

关于"压测要不要完全贴合线上"

结论：不能完全贴。过度拟合线上 = 低估未来负载风险（活动叠加、能力迭代）。压测必须并行设计：① 与线上一致的常规场景，② 单独的极端长上下文 / 极端多轮 / 工具调用专项，二者并行才能既保证今天的稳定性，也为未来留余量。

四、有效性校验四件套

  • 分布校验。token 区间、长尾比例与放量后的线上是否一致。
  • 数据质量。过滤无效 query、空上下文、异常样本。
  • 链路校验。所有 Throughput / 关键路径节点都被压到。
  • 回归校验。优化前后固定数据卡片，确保结果可比。

五、工程化沉淀 · 数据从哪来、往哪去

  • 来源混合：① 工程侧 / Agent 侧线上录制（保真，但低频场景捞不到、不可解释），② 手动按场景模板 + 物料拼接（可控、可解释、可补低频），二者混合。
  • 去向分层：反对一刀切的"全量仿真"——成本高、结论不可解释；改为分场景建模：每个子场景单独建一个负载模型，压测时按权重发压，线上权重变化时只改权重不重做数据。

六、四个核心结论

  1. 数据决定了压测是否可信。
  2. 压测数据构造的本质是负载建模，不是请求构造。
  3. 高质量数据要同时关注真实性、覆盖度、复杂度。
  4. 工程化沉淀比一次性造数更重要——一次造的数明天就过期。

七、现场问答精选

  • Q · 压测发现瓶颈在模型本身怎么处理？
    先验证负载是否真的是预期负载（先排压测自己的问题）；再看具体原因（如上下文未压缩导致前缀缓存未命中）；都不行就调外部参数；最后扩卡。
  • Q · 小红书强依赖大模型的功能场景有哪些？
    主站 AI 助手、评论辅助、点点端攻略 / 提醒、ASR、翻译等。
  • Q · 测试环境算力只有生产 1/10，能否线性外推峰值？
    不能。功能测试和压测先分开；压测要在和生产对齐配比的测试集群做，不建议线性外推。
  • Q · 有没有量化指标度量"压测和线上接近度"？
    分两段看：来源数据宽度（卡片类型、L2 类型、图片占比是否一致）；落地表现（前端缓存、新增明细、首包时间是否对齐）。

一、为什么端到端测试始终是金字塔的"沉没成本"

  • 三大特征：长域 / 长链路 / 组合爆炸。电商场景 5~6 个领域、新人手工造数 1~2 天、组合数指数级。
  • 传统解：原子化封装 + 流程编排（代码 / DSL / 拖拽）。早期很爽、后期腐化——底层封装仍依赖人，业务一抖动整套就要重写。
  • AI 时代的新选择：直接让 AI 感知原生服务接口，扔掉"人工预设流程"这一层。

二、测试执行 Agent 的目标能力

  • 动态编排：给一句自然语言诉求 / 用例文本，自动推导出"先调 A、再调 B、最后调 C"的执行计划。
  • 韧性执行：对接口变化、参数错误、异常返回有 retry / 自我修正能力。

三、整体架构

层组件作用
接入IDE / CLI / 测试同学网页多入口
规划 + 执行测试执行 AgentPlan → Q → 执行（宏观），Coding-Agent 框架（微观）
能力底座业务知识库领域 / 跨领域知识
接口列表每个域内接口的 Schema
跨环境网关跨号调用、安全管控、流量回放
脚本管理（Git）检索 / 分发 / 权限

认知框架混合：宏观用 Plan-Q（计划 → 校验 → 执行），微观用 Coding-Agent 框架。前者锁死业务目标不跑偏，后者保证执行细节的鲁棒性。

四、关键差异 · 测试 Agent ≠ Coding Agent

  • Coding Agent 通用能力强，但不懂业务：不知道"纠纷单 = 平台介入"，不熟悉上下游是支付还是结算系统，不熟悉 Schema 优先级。
  • 测试 Agent 必须做两件事：看懂业务 + 写好脚本。

五、看懂业务 · "动作—实体—属性"三元拆解 + 链式倒推

  • 把测试诉求拆成 Action（下单 / 发货 / 结算）/ Entity（订单 / 合约 / 结算单）/ Attribute（实体上的状态 / 值约束）。
  • 从"右端"（最终交易域）倒推：要验证买手对账金额 → 需要结算单 → 需要订单 → 需要合约 → 需要商品 / 用户 / 地址。
  • 意图模糊时主动澄清，避免大模型自由发挥。

六、知识库 · 故意不选 RAG

测试领域召回不全 = 漏测。RAG 适合海量信息相关性召回，但对召回完整性要求极高的测试 Plan 来说不合适。所以采用"按领域 / 按工具的按需加载"——Plan 阶段先加载交易域，识别到买手依赖再加载买手域，再到商品域，逐层拉 Schema，避免大模型瞎猜。

七、写脚本 · Debug-First + 流量回放

  • 有了 To-Do List → 先用向量召回历史脚本；命中即复用；不命中走 Debug-First：命令行逐个调接口，探索 input / output Schema，再生成脚本。
  • 用流量回放验证生成脚本——把脚本作为被测对象，用前序 Debug 阶段的 input / output 流量做回归。
  • 沉淀两类经验：① 工具级 tip（如 order_id 含分隔符前缀，需先剥版本号再传支付接口）；② 成功路径脚本作为下次召回的参考样本——10~20 个用例后新用例一次成功率显著上升。

八、Prompt 实践 · 不是技巧，是结构

  • Prompt 的本质是把团队对业务的"规范化描述"喂给模型——COT / Few-Shot 都是表层。
  • "问题拆得足够小、每步都有具体产出物"是稳态执行的关键。模型按填表方式输出，比让它自由发挥要稳定得多。
  • Agent 的另一关键是"记忆"——知识库 + 脚本经验双库联用，提供横向迁移与自我进化能力。

九、落地场景 · 数据构造

切入点选了"下游业务的造数 = 上游业务的自动化用例"——把造数解决掉，再上探到运营执行。当前在 10 多个电商类产品上：

  • 未命中缓存：分钟级完成造数；命中缓存：秒级。
  • 单次成本 ≈ ¥1.5 / $0.20；优化后预计可再降 20%+。
  • 真实演示："beta 环境购物车购买显卡，手机号 1234..." → Agent 自动拆 Action / Entity → 调度账号 / 地址 / 下单 / 支付链路 → 命中入参错误时自我修正 → 输出订单 + 商品 + 收货地址完整结果。
